### US-001: 新規プラグイン生成

**ユーザー**: プラグイン管理者（U1）として

**ストーリー**:
> 私は、ウィザードに従って自チームの情報を入力するだけで、すぐに使えるコンテキスト管理プラグインを生成したい。そうすることで、Markdownテンプレートの詳細を知らなくてもチーム専用のプラグインをゼロから作れる。

**実装根拠**:
- `commands/generate.md` — `/forge-generate` コマンド（引数なし）
- `skills/generate/SKILL.md` — 新規ウィザードモード
- `skills/generate/references/wizard-steps.md` — Step 1-5 の対話フロー

**受け入れ基準**:
- [ ] `/forge-generate` を引数なしで実行するとウィザードが起動する
- [ ] Step 1 でチーム名・事業名・プレフィクスを入力できる
- [ ] Step 2 でストレージ種別と接続情報を選択できる
- [ ] Step 3 で営業フレームワークを選択できる（デフォルト: BANTCH）
- [ ] Step 4 でデータソースを選択できる（デフォルト: Slack/GCal/Gmail/GDrive有効）
- [ ] Step 5 でナレッジカテゴリを設定できる（デフォルト: 2カテゴリ）
- [ ] 確認後に生成が実行され、`.plugin` ファイルが作成される
- [ ] 生成物に未解決変数（`{{...}}`）が含まれない

### US-002: 既存設定からの再生成

**ユーザー**: プラグイン管理者（U1）として

**ストーリー**:
> 私は、プレフィクスや商材名を変更したい場合、既存の設定ファイルを基に再生成したい。そうすることで、ウィザードを最初からやり直さなくても設定を部分的に更新できる。

**実装根拠**:
- `commands/generate.md` — 引数あり（config パス指定）分岐
- `SKILL.md` — 再生成モード処理
- `dataflow.md` — プラグイン再生成フロー

**受け入れ基準**:
- [ ] `/forge-generate path/to/.team-config.yml` で再生成モードが起動する
- [ ] 現在の設定内容が表示される
- [ ] 変更する項目のみ再入力できる
- [ ] 変更後の設定が `.team-config.yml` に保存される
- [ ] 再生成物に矛盾がない（新しいプレフィクスが全ファイルに反映される）

### US-005: 不要コマンドの除外

**ユーザー**: プラグイン管理者（U1）として

**ストーリー**:
> 私は、チームが使わないコマンド（例: `engdoc-*` や `doc-estimate`）を除外してすっきりしたプラグインを生成したい。そうすることで、不要なコマンドでユーザーが混乱せず、チームの実務に合ったプラグインを提供できる。

**実装根拠**:
- `config-schema.md` — `excluded_commands` フィールド定義
- `SKILL.md` Step 5 — excluded_commands フラグ適用
- `TC-019-excluded-commands.md` — 除外テスト

**受け入れ基準**:
- [ ] `.team-config.yml` の `excluded_commands` に指定したコマンドが除外される
- [ ] ファイル単位除外（`deal-load` 等）: 対応ファイルが生成されない
- [ ] セクション単位除外（`admin-backlog` 等）: 統合ファイルの該当セクションが非表示
- [ ] `admin-members` 除外時は 3ファイルが連動して更新される

### US-006: 生成物の品質確認

**ユーザー**: プラグイン管理者（U1）として

**ストーリー**:
> 私は、生成されたプラグインが壊れていないことを確認した上でパッケージングしてほしい。そうすることで、インストール後に「変数が展開されていない」等の問題を発見するリスクを減らせる。

**実装根拠**:
- `post-generation-check.md` — 6項目バリデーション定義
- `SKILL.md` Step 6 — 生成物チェック実行
- `TC-029-post-generation-validation.md`

**受け入れ基準**:
- [ ] 生成後に6項目のチェックが自動実行される
- [ ] チェック結果が統一フォーマットで表示される
- [ ] NG項目がある場合、パッケージングが実行されない
- [ ] NG時に修正すべき問題の詳細が表示される
- [ ] 全PASS時のみ `.plugin` ファイルが生成される

### US-007: 既存コンテキストデータのマイグレーション

**受け入れ基準**:
- [ ] `/forge-migrate path/to/plugin` でマイグレーションが起動する

- [ ] 処理対象（全件/カテゴリ別/N件）を選択できる
- [ ] マイグレーション完了後に成功/失敗/スキップ件数が報告される

### US-008: テンプレート改善後の整合性確認

**ユーザー**: コントリビューター（U2）として

**ストーリー**:
> 私は、テンプレートファイルを変更した後、整合性が壊れていないことを体系的に確認したい。そうすることで、変更による意図しない破壊（未解決変数・参照切れ等）をコミット前に発見できる。

**実装根拠**:
- `templates/forge-consistency-check-prompt.md` — 10項目チェック定義
- `CLAUDE.md` — 整合性チェック必須ルール
- `TC-046-consistency-check.md`

**受け入れ基準**:
- [ ] 10項目の整合性チェックリストに従って確認できる
- [ ] 各チェック項目がPASS/NGで報告される
- [ ] テンプレート変更の種類に応じて必要なチェック項目がわかる
- [ ] 推測・省略なしのファイル読み込みによる事実ベースのチェックができる
